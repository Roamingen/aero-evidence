# 需求文档

最后更新: 2026-03-15 (下午)

## 1. 项目目标
构建一个面向民航客机检修记录的存证系统，采用“链下业务数据 + 链上关键摘要存证”的架构，支持检修记录提交、多人签名审核、驳回重提、防篡改校验和权限化访问。

## 2. 范围说明
本项目当前确定采用以下技术路线：
- 前端: Vue3 + Vite
- 后端: Node.js + Express + ethers
- 区块链: 开发阶段使用 Hardhat 本地链，部署阶段使用 Hyperledger Besu（QBFT）
- 数据库: MySQL
- 智能合约: Solidity 0.8.24

当前阶段明确不做的内容：
- 复杂工作流引擎
- 消息队列和微服务拆分
- 正式对象存储平台治理
- 企业级 CA / 国密电子签章体系

## 4. 业务规则
- 草稿、自动保存、填写错误的中间版本不上链。
- 正式提交时才上链写入该 revision 的业务摘要和附件 manifest 摘要。
- 技术会签、审核会签、RII 批准、最终放行等关键里程碑再追加链上签名。
- 驳回后不覆盖旧记录，必须生成新 revision 重提，保留完整历史链路。
- 签名人地址必须与当前登录用户绑定地址一致。

## 5. 非功能需求
- 安全性: 私钥仅保留在用户钱包或前端本地受控环境中，不在后端落盘。
- 可维护性: 后端采用 routes / controllers / services / models 分层，区块链接入与业务编排分离。
- 可部署性: 支持本地 Hardhat 开发联调，支持云端 Besu QBFT 部署。
- 可扩展性: 支持从单节点开发环境扩展到多节点联盟链。

## 6. 当前确认的技术决策
- 已放弃 FISCO BCOS 作为当前实现路径。
- 当前主线为 Hyperledger Besu + Hardhat + Node.js + ethers。
- 开发联调与生产部署共用 Solidity / EVM 合约能力，避免双栈维护。
- Hardhat 3 通过 `backend/hardhat-local` 子项目隔离，避免和后端 CommonJS 冲突。
- 合约当前编译启用 `viaIR`，用于解决 Solidity `stack too deep` 编译限制。
- 当前签名摘要依赖 `recordId`，因此首次提交必须由前端先生成 `recordId`，驳回重提必须由前端先生成 `nextRecordId`。

# 接口文档

最后更新: 2026-03-08

## 1. 基本说明

### 基础地址
- 本地开发: `http://127.0.0.1:3000`

### 响应格式
- 成功: 返回 JSON 对象，结构由各接口决定。
- 失败: 统一返回

```json
{
  "message": "错误说明"
}
```

### 鉴权方式
- 未特别说明的接口为公开接口。
- 需要登录的接口使用 HTTP Header:

```text
Authorization: Bearer <jwt-token>
```

### 常见状态码
- `200`: 请求成功
- `201`: 创建成功
- `400`: 参数错误或摘要不一致
- `401`: 未登录、签名错误、签名地址不匹配
- `403`: 权限不足或账户状态不可用
- `404`: 资源不存在
- `409`: 业务冲突，例如重复激活、重复重提
- `410`: 激活码或 challenge 已过期
- `423`: 激活码被锁定
- `500`: 服务器内部错误

## 3. 认证接口

### `POST /api/auth/admin/preregister`

#### 说明
- 管理员预注册员工账户。
- 当前通过 Header `x-admin-bootstrap-key` 保护。

#### 请求头

```text
x-admin-bootstrap-key: <bootstrap-key>
Content-Type: application/json
```

#### 请求体

```json
{
  "employeeNo": "E1001",
  "name": "张三",
  "department": "维修一部",
  "roleCodes": ["engineer_submitter"],
  "deliveryChannel": "manual"
}
```

#### 字段说明
- `employeeNo`: 必填，工号。
- `name`: 必填，姓名。
- `department`: 选填，部门。
- `roleCodes`: 选填，角色编码数组。
- `deliveryChannel`: 选填，激活码发放方式。

#### 成功响应

```json
{
  "user": {
    "id": 1,
    "employeeNo": "E1001",
    "name": "张三",
    "department": "维修一部",
    "status": "pending_activation",
    "roles": ["engineer_submitter"],
    "permissions": []
  },
  "activationCode": "123456",
  "activationCodeLast4": "3456",
  "activationCodeExpiresAt": "2026-03-08T10:00:00.000Z"
}
```

## 4. Maintenance 接口

### 共同说明
- 以下接口均需要 JWT。
- 当前提交摘要使用以下结构计算：

```json
{
  "recordId": "0x...",
  "action": "submit|technician_sign|reviewer_sign|rii_approve|release|reject|revoke",
  "formHash": "0x...",
  "attachmentManifestHash": "0x...",
  "signerEmployeeNo": "E1001"
}
```

- 产品前端默认不手填 `recordId`、`jobCardNo`、执行人信息。
- 正式提交流程应先调用 `POST /api/maintenance/records/prepare`，再对返回的 `signedDigest` 做本地签名。

#### 说明
- 查询检修记录列表，供查阅中心使用。
- 支持参数: `page`、`pageSize`、`status`、`keyword`、`aircraftRegNo`。

  "pageSize": 8,
  "total": 12,

- `specifiedSigners`: 选填，若传入则后续对应角色签名必须命中名单。
- `manifest.attachments`: 必须为数组。
- `signedDigest`: 必填，必须与服务端重算结果一致。
- `signature`: 必填，签名地址必须与当前 JWT 对应地址一致。

### `POST /api/maintenance/records/:recordId/signatures`

#### 说明
- 对既有记录追加业务签名。

#### 请求体示例

```json
{
  "signerRole": "reviewer",
  "action": "reject",
  "rejectionReason": "资料描述需要补充",
  "signedDigest": "0x...",
  "signature": "0x...",
  "signaturePayloadPath": "signatures/reject-E2001.json"
}
```

#### `signerRole` 可选值
- `technician`
- `reviewer`
- `rii_inspector`
- `release_authority`
- `system_node`

#### `action` 可选值
- `technician_sign`
- `reviewer_sign`
- `rii_approve`
- `release`
- `reject`
- `revoke`

#### 业务规则
- 同一地址不能对同一动作重复签名。
- `release` 前必须满足技术签名数和审核签名数门槛。
- `resubmit` 不是这里的 action，而是单独接口。

#### 成功响应
- 返回更新后的完整记录详情对象。

## 5. 当前限制
- maintenance 列表接口尚未实现。
- maintenance 接口当前要求客户端自己生成 `recordId` / `nextRecordId`。
- `signerRole` 与 `action` 还没有完全接到正式 RBAC 权限矩阵。
- 防篡改校验接口尚未开放。

## 6. 调用顺序建议

### 认证流程
1. `POST /api/auth/admin/preregister`
2. `POST /api/auth/activate/challenge`
3. 钱包对 challenge 签名
4. `POST /api/auth/activate/verify`
5. `POST /api/auth/nonce`
6. 钱包对登录 challenge 签名
7. `POST /api/auth/verify`
8. 后续请求带 `Authorization: Bearer <token>`

### maintenance 提交流程
1. 前端生成 `recordId`
2. 前端按规则准备业务数据
3. 前端计算待签摘要并签名
4. 调用 `POST /api/maintenance/records`
5. 后续按需要调用签名接口或重提接口

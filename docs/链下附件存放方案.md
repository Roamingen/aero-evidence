# 链下附件存放方案

最后更新: 2026-03-07

## 1. 基本原则
- 附件文件本体不进数据库，不上链。
- 数据库存附件元数据、哈希、存放路径、处理状态。
- 链上只保存附件清单摘要 `attachmentManifestHash` 和数量统计。
- 本地开发阶段统一由后端管理附件目录，前端只上传，不能直接决定最终存放路径。

## 2. 本地目录建议
本地开发与单机部署阶段，附件统一放在后端目录下：

```text
backend/storage/
  attachments/
    2026/
      B-1234/
        REC-20260307-0001/
          manifest.json
          documents/
            DOC-001-job-card.pdf
            DOC-002-maintenance-report.docx
          images/
            IMG-001-fault-photo.jpg
            IMG-002-after-repair.jpg
          videos/
            VID-001-inspection.mp4
          derived/
            IMG-001-thumb.jpg
            VID-001-preview.mp4
  tmp/
  quarantine/
```

目录分层建议：
- 第一层按年份，例如 `2026/`
- 第二层按飞机注册号，例如 `B-1234/`
- 第三层按记录号，例如 `REC-20260307-0001/`
- 记录目录下再按 `documents/`、`images/`、`videos/`、`derived/` 分类

这样做的目的：
- 方便按记录整体迁移和打包
- 方便一个记录下多个附件集中审计
- 方便以后把本地磁盘切换到 NAS 或对象存储

## 3. 数据库存什么
数据库需要保存：
- 记录 ID
- 附件 ID
- 附件类型
- 原始文件名
- MIME 类型
- 文件大小
- 内容哈希 `content_hash`
- 缩略图哈希 `thumbnail_hash`
- 存储介质 `storage_disk`
- 存储路径 `storage_path`
- 预览路径 `preview_path`
- 转码路径 `transcoded_path`
- 上传人、上传时间、状态

数据库不保存：
- 文件二进制内容
- 大图片本体
- 视频流本体

## 4. 清单 manifest 怎么做
每条检修记录维护一份 `manifest.json`，示例：

```json
{
  "recordId": "REC-20260307-0001",
  "version": 1,
  "attachments": [
    {
      "attachmentId": "DOC-001",
      "type": "document",
      "fileName": "job-card.pdf",
      "mimeType": "application/pdf",
      "size": 245678,
      "contentHash": "0x1234...",
      "storagePath": "2026/B-1234/REC-20260307-0001/documents/DOC-001-job-card.pdf"
    },
    {
      "attachmentId": "IMG-001",
      "type": "image",
      "fileName": "fault-photo.jpg",
      "mimeType": "image/jpeg",
      "size": 845221,
      "contentHash": "0xabcd...",
      "storagePath": "2026/B-1234/REC-20260307-0001/images/IMG-001-fault-photo.jpg"
    }
  ]
}
```

后端流程：
- 上传完成后生成或更新 `manifest.json`
- 对 `manifest.json` 做规范化序列化
- 计算 `manifestHash`
- 将 `manifestHash` 同步到数据库和合约

## 5. 上传处理流程
建议的上传流程：

1. 前端上传文件到后端临时目录 `backend/storage/tmp/`
2. 后端计算文件哈希并做基础校验
3. 可选执行病毒扫描、图片缩略图生成、视频转码
4. 处理成功后移动到正式目录 `backend/storage/attachments/...`
5. 更新 `maintenance_attachments` 和 `maintenance_attachment_manifests`
6. 重新计算 `attachmentManifestHash`
7. 如需上链，再调用合约更新附件摘要

## 6. 本地开发阶段怎么管理
本地 MVP 阶段建议：
- 存储介质固定为 `local`
- 文件根目录固定为 `backend/storage/attachments`
- 每个附件保存原始文件，不做复杂分布式存储
- 图片可以额外生成缩略图到 `derived/`
- 视频可以暂时不转码，只保留原文件路径和可选预览图

这不等于“裸奔”。当前最小原型阶段的控制点仍然是：
- 只有后端服务能决定最终存放路径
- 数据库存储哈希、状态和审计信息
- 合约存储附件清单摘要，支持后续校验

现阶段不必一上来就上 Docker 附件服务，原因是：
- 你现在是单机开发阶段，本地磁盘最简单、调试成本最低
- 附件处理链路还没稳定，先把文件组织、哈希、manifest 逻辑做对更重要
- 以后切到 MinIO、NAS、S3 时，数据库字段和链上摘要模型都不用重做

适合引入 Docker/对象存储服务的时机是：
- 需要多人共享同一套附件仓库
- 需要和后端服务分机部署
- 需要更强的备份、扩容和权限控制
- 需要视频转码、病毒扫描、缩略图等独立处理服务

## 7. 以后怎么迁移
后续如果切到 NAS、MinIO、S3：
- 数据库结构不用大改
- 只需把 `storage_disk` 从 `local` 改为 `nas/minio/s3`
- `storage_path` 改成对象键或远程路径
- `attachmentManifestHash` 逻辑不变

这就是为什么 V2 结构里应该把附件做成“链下文件 + 数据库元数据 + 链上 manifest hash”的三层模型。

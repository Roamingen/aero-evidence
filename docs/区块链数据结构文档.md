## 快速对比：比特币 vs Aero Evidence

### 比特币区块结构

```
┌─────────────────────────────────────────┐
│          Bitcoin Block Header (80 字节)  │
├─────────────────────────────────────────┤
│ version (4B)                             │
│ previous block hash (32B)                │
│ merkle root (32B)  ◄── 交易树根          │
│ timestamp (4B)                           │
│ difficulty target (4B)                   │
│ nonce (4B)  ◄── PoW 随机数              │
├─────────────────────────────────────────┤
│ Transaction List (不定长)                │
│ - Input: 前一个 UTXO 的引用              │
│ - Output: 新 UTXO                        │
│ - Signature: 交易签名                    │
├─────────────────────────────────────────┤
│ Merkle Tree (验证交易完整性)             │
└─────────────────────────────────────────┘
```

**特点**：
- ✅ 链式结构 (Block n → Block n+1)
- ✅ 交易内容完全上链
- ✅ PoW 共识
- ❌ 存储低效 (每个完整区块 > 1MB)

### Aero Evidence 区块结构

```
┌─────────────────────────────────────────┐
│   Hyperledger Besu Block (与 Ethereum   │
│              相同结构)                   │
├─────────────────────────────────────────┤
│ Block Header:                            │
│ ├─ number: 区块序号                     │
│ ├─ timestamp: 时间戳                    │
│ ├─ parentHash: 前一个区块哈希           │
│ ├─ stateRoot: 状态树根                 │
│ ├─ transactionsRoot: 交易根            │
│ ├─ receiptsRoot: 收据根                │
│ ├─ miner: 出块者                       │
│ └─ difficulty/gasLimit: PoA 参数        │
├─────────────────────────────────────────┤
│ Transactions (合约调用):                 │
│ ├─ submitRecord(                        │
│ │  ├─ recordId (bytes32)                │
│ │  ├─ RecordCore { 基本信息 }           │
│ │  ├─ RecordDigest { 6个哈希值 }        │
│ │  ├─ AttachmentSummary { 附件摘要 }    │
│ │  ├─ signature { 签名证明 }            │
│ │  └─ ... [其他字段]                    │
│ │  )                                    │
│ └─ appendSignature(...) [后续签名]       │
├─────────────────────────────────────────┤
│ Receipts (执行结果):                     │
│ ├─ transactionHash                      │
│ ├─ status: 成功/失败                   │
│ ├─ blockNumber                          │
│ ├─ events: RecordSubmitted, StatusChanged
│ └─ gasUsed                              │
└─────────────────────────────────────────┘
```

**特点**：
- ✅ 离链/在链分离 (数据哈希上链，详细数据离链)
- ✅ 高效存储 (仅 32 字节哈希)
- ✅ PoA 共识 (无 PoW 计算)
- ✅ 智能合约 (可编程验证)

## 完整数据结构

### 1. 智能合约中的核心数据结构

#### RecordCore (记录核心)

```solidity
struct RecordCore {
    bytes32 recordId;                         // 记录唯一标识
    string aircraftRegNo;                     // 飞机注册号 (B-4321)
    string aircraftType;                      // 机型 (A320-200)
    string jobCardNo;                         // 工卡号 (AUTO-20260317-ABC12345)
    uint32 revision;                          // 修订号 (1, 2, 3...)
    string ataCode;                           // ATA 章节 (32-11)
    string workType;                          // 工作类型 (功能检查、维修等)
    string locationCode;                      // 维修位置 (HGH-H2)
    string performerEmployeeNo;               // 执行人工号 (E1001)
    uint16 requiredTechnicianSignatures;      // 需要的技术签名数
    uint16 requiredReviewerSignatures;        // 需要的审核签名数
    uint16 technicianSignatureCount;          // 当前技术签名数
    uint16 reviewerSignatureCount;            // 当前审核签名数
    bool isRII;                               // 是否需要 RII 检查
    uint64 occurrenceTime;                    // 发生时间 (Unix 时间戳, 秒)
    uint64 createdAt;                         // 创建时间 (区块时间)
    uint64 updatedAt;                         // 最后更新时间
    RecordStatus status;                      // 当前状态 (enum)
}
```

**字段映射**：

| 字段 | 类型 | 长度 | 用途 | 示例 |
|-----|------|------|------|------|
| recordId | bytes32 | 32字节 | 唯一键，由前端生成 | `0x8a1b2c...` |
| aircraftRegNo | string | 可变 | 飞机注册号 | `"B-4321"` |
| jobCardNo | string | 可变 | 工卡号 | `"AUTO-20260317-ABC12345"` |
| revision | uint32 | 4字节 | 修订号，拒绝后重新提交 | `1` |
| performerEmployeeNo | string | 可变 | 提交人工号 | `"E1001"` |
| occurrenceTime | uint64 | 8字节 | 故障发生时间 | `1710748200` |
| createdAt | uint64 | 8字节 | 区块时间 | 自动填充 |
| status | enum | 1字节 | 记录状态 | `1 = Submitted` |

#### RecordDigest (哈希摘要)

```solidity
struct RecordDigest {
    bytes32 formHash;                         // 基础表单哈希
    bytes32 faultHash;                        // 故障信息哈希
    bytes32 partsHash;                        // 部件清单哈希
    bytes32 measurementsHash;                 // 测量项目哈希
    bytes32 replacementsHash;                 // 替换项目哈希
    bytes32 attachmentManifestHash;           // 附件清单哈希
}
```

**验证逻辑**：
```
离链数据 → 规范化 → Keccak-256(canonical JSON) → bytes32 hash
                                                   ↓
                                          存储在智能合约中
                                                   ↓
                                          验证时重新计算对比
```

#### SignatureProof (签名证明)

```solidity
struct SignatureProof {
    SignatureRole signerRole;                 // 签名角色 (enum 0-4)
    SignatureAction action;                   // 动作 (enum 1-6)
    address signer;                           // 签名者钱包地址
    string signerEmployeeNo;                  // 签名者工号
    bytes32 signedDigest;                     // 被签名的摘要
    bytes32 signatureHash;                    // 签名本身的 Keccak-256
    uint64 signedAt;                          // 签名时间
}
```

**SignatureRole (枚举)**：
```
0 = Technician (技术签名人)
1 = Reviewer (审核签名人)
2 = RiiInspector (RII 检查员)
3 = ReleaseAuthority (放行授权人)
4 = SystemNode (系统节点)
```

**SignatureAction (枚举)**：
```
1 = TechnicianSign (技术签名)
2 = ReviewerSign (审核签名)
3 = RiiApprove (RII 批准)
4 = Release (放行)
5 = Reject (拒绝)
6 = Revoke (撤销)
```

---

#### RecordStatus (状态枚举)

```solidity
enum RecordStatus {
    Draft,          // 0 - 草稿
    Submitted,      // 1 - 已提交（等待第一个技术签名）
    PeerChecked,    // 2 - 同行评审中（已获得技术签名，等待审核）
    RiiApproved,    // 3 - RII 已批准（等待放行）
    Released,       // 4 - 已放行（完成）
    Rejected,       // 5 - 已拒绝（需要重新提交）
    Revoked         // 6 - 已撤销（管理员操作）
}
```

**状态转移图**：
```
Draft
  ↓
Submitted (1个技术签名)
  ↓
PeerChecked (N个审核签名)
  ↓
RiiApproved (如果 isRII=true)
  ↓
Released (1个放行签名) ✅ 完成
  ↗
  Rejected → [重新提交] → Submitted

Revoked (任何状态下，管理员可撤销)
```

## 离链数据存储

### MySQL 数据库架构

#### 主表：maintenance_records

```sql
CREATE TABLE maintenance_records (
    -- 主键和关联
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    record_id CHAR(66) UNIQUE NOT NULL,           -- 0x... (32 字节 hex)
    root_record_id CHAR(66) NOT NULL,             -- 初始记录 ID
    previous_record_id CHAR(66),                  -- 前一个版本
    superseded_by_record_id CHAR(66),             -- 被后续版本替代

    -- 基础字段
    aircraft_reg_no VARCHAR(50) NOT NULL,
    aircraft_type VARCHAR(100) NOT NULL,
    job_card_no VARCHAR(100) UNIQUE NOT NULL,
    revision INT DEFAULT 1,
    ata_code VARCHAR(20) NOT NULL,
    work_type VARCHAR(100) NOT NULL,
    location_code VARCHAR(50),

    -- 人员信息
    performer_user_id BIGINT NOT NULL,
    performer_employee_no VARCHAR(50) NOT NULL,
    performer_name VARCHAR(100),

    -- 签名要求
    required_technician_signatures INT DEFAULT 1,
    required_reviewer_signatures INT DEFAULT 1,
    technician_signature_count INT DEFAULT 0,
    reviewer_signature_count INT DEFAULT 0,
    is_rii BOOLEAN DEFAULT FALSE,

    -- 时间信息
    occurrence_time DATETIME NOT NULL,
    submitted_at DATETIME,
    rejected_at DATETIME,
    resubmitted_at DATETIME,
    released_at DATETIME,

    -- 状态
    status ENUM('submitted', 'peer_checked', 'rii_approved', 'released', 'rejected', 'revoked') DEFAULT 'submitted',
    rejection_reason TEXT,

    -- 哈希值
    form_hash CHAR(66),
    fault_hash CHAR(66),
    parts_hash CHAR(66),
    measurements_hash CHAR(66),
    replacements_hash CHAR(66),
    attachment_manifest_hash CHAR(66),

    -- 链上信息
    chain_record_id CHAR(66),                     -- 对应的链上 recordId
    chain_tx_hash CHAR(66),                       -- 提交交易 hash
    chain_block_number BIGINT,                    -- 所在块高

    -- 审计
    created_by BIGINT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    KEY idx_aircraft_reg_no (aircraft_reg_no),
    KEY idx_job_card_no (job_card_no),
    KEY idx_performer_employee_no (performer_employee_no),
    KEY idx_status (status),
    KEY idx_chain_record_id (chain_record_id)
);
```

#### 签名表：maintenance_record_signatures

```sql
CREATE TABLE maintenance_record_signatures (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    record_id BIGINT NOT NULL REFERENCES maintenance_records(id),

    -- 签名信息
    signer_role ENUM('technician', 'reviewer', 'rii_inspector', 'release_authority') NOT NULL,

    signer_user_id BIGINT NOT NULL,
    signer_employee_no VARCHAR(50) NOT NULL,
    signer_name VARCHAR(100),
    signer_address VARCHAR(42) NOT NULL,         -- 0x... (20 字节 hex)

    -- 签名摘要

    signature_hash CHAR(66) NOT NULL,            -- Keccak-256(signature)

    signature_payload_path VARCHAR(255),         -- 可选的有效负荷路径

    -- 链上信息

    signed_at DATETIME NOT NULL,

    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    KEY idx_record_id (record_id),
    KEY idx_signer_employee_no (signer_employee_no),
    KEY idx_signed_at (signed_at)
```

## 数据流向

### 完整的提交流程数据流

```
┌─────────────────────────────────────────────────────────────────┐
│                      前端 (Vue + ethers.js)                     │
│                                                                 │
│  用户填写表单 → 上传文件 → 计算哈希 → 请求签名前上下文         │
│        ↓          ↓          ↓              ↓                   │
│   表单数据    [contentHash]  [cHash, tHash]  [请求体]          │
└────────────────────────────┬──────────────────────────────────┘
                             │ HTTP POST /api/maintenance/records/prepare
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                   后端 (Node.js Express)                        │
│                                                                 │
│  规范化数据 → 计算6个哈希 → 合成signedDigest → 返回           │
│      ↓           ↓            ↓                  ↓             │
│  [JSON trim]  [Keccak256]  [recordId +    [signedDigest]      │
│                             action + ...]                      │
│                                                                 │
│  MySQL 状态: 暂不存储 (待提交)                                 │
└────────────────────────────┬──────────────────────────────────┘
                             │ 返回 JSON response
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                      前端 (Vue + ethers.js)                     │
│                                                                 │
│  接收 signedDigest → 用私钥签名 → 提交记录                    │
│        ↓                ↓              ↓                        │
│  [bytes32]      [signature]   [POST /api/maintenance/records]  │
│                                                                 │
│  发送内容:                                                      │
│  {                                                              │
│    recordId, aircraftRegNo, ..., signedDigest, signature       │
│  }                                                              │
└────────────────────────────┬──────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                   后端 (Node.js Express)                        │
│                                                                 │
│  ① 验证签名:                                                   │
│     ethers.verifyMessage(signedDigest, signature)             │
│     → 恢复的地址 == 当前用户地址? ✓                           │
│                                                                 │
│  ② 重新计算 signedDigest:                                      │
│     expectedDigest = hashJson({...})                          │
│     expectedDigest == 前端的 signedDigest? ✓                 │
│                                                                 │
│  ③ 存储到 MySQL:                                              │
│     INSERT INTO maintenance_records                           │
│     (record_id, job_card_no, ..., chain_tx_hash, status)     │
│     INSERT INTO maintenance_record_signatures                │
│     (record_id, signer_role, signed_digest, signature_hash)  │
│     INSERT INTO attachments                                  │
│     (record_id, file_name, content_hash, ...)                │
│                                                                 │
│  ④ 调用智能合约:                                               │
│     contract.submitRecord(input, signature)                  │
│     → 获得 txHash, blockNumber                               │
│                                                                 │
│  ⑤ 更新链上信息:                                               │
│     UPDATE maintenance_records                               │
│     SET chain_record_id = recordId,                          │
│         chain_tx_hash = txHash,                              │
│         chain_block_number = blockNumber,                    │
│         status = 'submitted'                                 │
└────────────────────────────┬──────────────────────────────────┘
                             │ 返回成功响应
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                      前端 (Vue + ethers.js)                     │
│                                                                 │
│  显示成功消息，展示:                                           │
│  - recordId                                                    │
│  - chainTxHash                                                │
│  - jobCardNo                                                  │
│  - 状态: submitted                                            │
└─────────────────────────────────────────────────────────────────┘
```

## 存证和验证

### 完整性验证流程

```
目标: 证明 "离链数据未被篡改"

① 获取链上的 formHash 和 attachmentManifestHash

② 在离链数据上重新计算:
   localFormHash = Keccak256(canonicalJson({
     aircraftRegNo, aircraftType, ..., workDescription
   }))

   localAttachmentHash = Keccak256(canonicalJson({
     version, generatedAt, attachments[...]
   }))

③ 对比:
   if (localFormHash == chainFormHash) {
     ✓ 基础信息未被篡改
   }
   if (localAttachmentHash == chainAttachmentHash) {
     ✓ 附件清单未被篡改
   }
   if (both match) {
     ✓ 整条记录完整、未被篡改
   }
```

### 签名验证流程

```
目标: 证明 "这个签名来自于该用户"

① 从链上获取:
   - signedDigest (被签名的摘要)
   - signature (签名本身)
   - signer (签名者地址)

② 恢复签名地址:
   recoveredAddress = ethers.verifyMessage(
     ethers.getBytes(signedDigest),
     signature
   )

③ 对比:
   if (recoveredAddress == signer) {
     ✓ 签名真实，来自该地址
   }
```
